AmigaSource — Development Roadmap

Phase-by-phase plan  ·  Custom admin module  ·  Multi-contributor support
Client: amigasource.com  |  Stack: PHP + MySQL + HTML + single CSS (no frameworks)  |  Target browser: IBrowse 3.0 (AmigaOS)  |  Drafted: June 2026, updated July 2026

Project Overview

The site has a working public front-end (browse + basic search) and a third-party admin (Maestro) that the client dislikes — it "works but has a different feel than the rest of the site." The goal is to replace Maestro with a lean, custom-built admin portal that matches the site's look and feel, supports multiple contributors, and progressively adds user-facing features without disrupting the clean single-page experience visitors already have.

All code must stay in single-file PHP pages, one shared CSS file, and vanilla PHP/MySQL — no Composer, no npm, no build tools, no frameworks. Modular, readable, maintainable. Old-school done properly.

Hard constraint confirmed July 2026: the client's real target browser is IBrowse 3.0 on AmigaOS (via AmiKit XE/WinUAE), not a modern desktop browser. IBrowse 3.0 has zero CSS support (per its own vendor FAQ: "no CSS yet, and only limited DOM support") and an old-but-real JavaScript engine (JS 1.6 / ECMA-262 rev3, ~2006-era) that predates fetch(), JSON.parse, and querySelectorAll. In practice this means the "single CSS file" and any public-facing JavaScript are cosmetic/no-ops for this client — actual rendering on their browser depends entirely on inline bgcolor/<font>/table-attribute markup. Every public-facing page and phase in this plan must be built and verified against that reality, not just against Chrome/Firefox.

Because this is someone else's codebase, a full audit phase is added before any changes. The order of phases is fixed: understand → stabilize → build. Do not skip Phase 00.

PHP 7.4+ MySQL / MariaDB HTML 4.01 / XHTML 1.0 Single CSS File (modern browsers only) No Framework No Composer No npm / Build Step No JavaScript Libraries No CSS Reliance (IBrowse 3.0 has none) No Public-Facing JS (IBrowse JS 1.6 only, admin-only exception)

Complexity Stars ★

★☆☆☆☆ Read-only display, no DB writes, no logic
★★☆☆☆ Basic CRUD on a single table
★★★☆☆ Multi-table ops, validation, business logic
★★★★☆ Auth/sessions, multi-actor workflows, or touching shared base code
★★★★★ Architecture-level decisions, major refactor risk, social features

MED Touches existing PHP or DB columns — regression-test after
HIGH Schema migration or shared foundation change — backup first, always

Picking Up Someone Else's Codebase — Checklist

Before Writing a Single Line

Run the site locally first. If it doesn't run, that is your first bug — fix it before touching anything else.
Read every file fully before judging anything. What looks wrong might be load-bearing.
Dump the full DB schema — print it out, read it, sketch the table relationships by hand.
Find the oldest file. It tells you how the original author thought and what constraints they were working under.
Search every $_GET, $_POST, $_REQUEST — these are the public attack surface. Note each one.
Search for raw SQL string concatenation — anything like "SELECT * WHERE id=" . $id is a SQL injection risk.
Look for hardcoded DB credentials or API keys inside PHP files. Move them to config.php before anything else.
Check for old mysql_* functions (not mysqli or PDO). These are removed in PHP 7+ and need replacing.

While Refactoring

Change one thing at a time. Test after each change. Never batch multiple refactors into one untested block.
Keep a plain CHANGES.txt log — what you changed, why, and when. Invaluable when something breaks two days later.
Rename things to mean what they currently do — not what the original author hoped they would do.
If you cannot figure out why a piece of code exists, do not delete it. Comment it with today's date and revisit after the full audit.
Only extract shared code to an include file after you have seen it duplicated in three or more places.
Do not change behavior while refactoring. Refactor first. Add features second. Never mix the two in the same session.
Every DB query must use PDO prepared statements. No exceptions — including internal admin-only queries.

Security — Non-Negotiable Before Launch

Every form input and URL parameter must be validated and sanitized before touching the DB or being echoed to the page.
Every variable echoed into HTML must go through htmlspecialchars($val, ENT_QUOTES, 'UTF-8') — no exceptions.
Admin pages: session authentication check must be the very first thing in every single admin PHP file.
File uploads (Phase 07): validate MIME type server-side — never trust the browser's reported content-type.
Never expose raw DB error messages to public users. Log to file, show a generic "something went wrong" message on screen.
Use password_hash() and password_verify() for admin passwords. No MD5. No plain text. No SHA1.
Set PHP session options: session.cookie_httponly = 1, session.use_strict_mode = 1 before session_start().

Communication with Client

After Phase 00, show the client the audit findings before starting Phase 01. They may know context that explains "bad" code.
Agree on naming conventions in writing. Put them in a CONVENTIONS.md file in the repo root. Reference it whenever you disagree.
Demo after every phase. Never go longer than 10 days without the client seeing working code in their browser.
Write one-liner comments in the code for any decision that will look strange to the next person who reads it.
